`Throughout the paper, the order is fixed.`

5. **a) Tytuł**
**Main theorem**

**b) Cele slajdu**
- Pokazać główny wynik wcześnie i bardzo wyraźnie.
- Ustawić wszystko, co dalej będzie wyjaśniane jako proof idea tego jednego twierdzenia.
- Uczciwie zaznaczyć różnicę między pamięcią dynamiczną a statycznym wynikiem bitowym.

**c) Treść slajdu**
- Centralny theorem box:
\[
\textbf{Theorem 1.}
\]
- `Let \(d\in\mathbb N\) and \(M\in\{0,1\}^{n\times n}\) be updated over time, always remaining \(d\)-twin-ordered.`
- `There is a dynamic data structure using \(O_d(n)\) memory and supporting:`
- `Init(n,\mathcal K)` in `\(O_d((n+|\mathcal K|)\log\log n)\)`
- `QueryBit(i,j)` in `\(O(\log\log n)\)` expected worst-case time
- `Update(i,j)` in `\(O(\log\log n)\)` expected worst-case time
- Dwa krótkie dopiski pod boxem:
- `\(\mathcal K\)` is a list of disjoint rectangles representing all ones
- `word RAM: \(O_d(n)\) memory means \(O_d(n\log n)\) bits`
- Jedna mała linia na dole:
`Only initialization and memory depend on \(d\).`

**d) Bardzo precyzyjny opis rysunku**
- Bez osobnej figury narracyjnej; głównym obiektem ma być theorem box.
- Pod theorem boxem mała tabelka `static vs dynamic` z trzema wierszami: `space`, `query`, `update`.
- W wierszu `space` uczciwie wpisać `static: \(O_d(n)\) bits`, `dynamic: \(O_d(n)\) memory`.
- W wierszu `update` komórka dla wyniku statycznego ma zawierać kreskę, a dla nowego wyniku pogrubione `\(O(\log\log n)\)`.

**e) Zakotwiczenie źródłowe**

10. **a) Tytuł**
**Amortized structure: \(\mathbb P\) for slabs, \(\mathbb Q\) for recent flips**

**b) Cele slajdu**
- Przejść od reprezentacji do właściwej dynamicznej struktury danych.
- Wyjaśnić dwuwarstwowy inwariant bez jeszcze rozbierania przebudowy.
- Uczynić `QueryBit` i `Update` intuicyjnymi na jednym obrazie.

**c) Treść slajdu**
- Dwa duże boksy:
- `\(\mathbb P\)`: static 2D orthogonal point location on the current canonical slab decomposition, storing at most `\(4f_d(n+2)+4n\)` slabs
- `\(\mathbb Q\)`: hash table of recent updates
- Próg bufora:
\[
|\mathbb Q| \le 8f_d(n+2)
\]
- Inwariant:
\[
(i,j)\in \mathbb Q \Rightarrow M[i,j]=\mathbb Q[i,j]
\]
\[
(i,j)\notin \mathbb Q \Rightarrow M[i,j]=1 \iff (i,j)\text{ lies in a slab stored in }\mathbb P
\]
- Dwa krótkie punkty:
- `QueryBit`: check `\(\mathbb Q\)` first, otherwise issue a point-location query to `\(\mathbb P\)`
- `Update`: toggle the value in `\(\mathbb Q\)`; rebuild only when the threshold is reached

**d) Bardzo precyzyjny opis rysunku**
- Jedna macierz jako tło.
- Kilka dużych półprzezroczystych prostokątów oznaczonych `stored in \(\mathbb P\)`.
- Na wierzchu 3–4 pojedyncze komórki z grubym obramowaniem oznaczone `stored in \(\mathbb Q\)`.
- Po lewej mała strzałka `QueryBit(i,j): \(\mathbb Q\) first, then \(\mathbb P\)`.
- Po prawej mała strzałka `Update(i,j): only \(\mathbb Q\) for now`.

12. **a) Tytuł**
**From amortized to worst-case: epochs and snapshot copies**

**b) Cele slajdu**
- Domknąć ostatni brak: dlaczego `Update` staje się worst-case, a nie tylko amortized.
- Pokazać prostą ideę epok i catch-upu.
- Wyjaśnić, po co naprawdę potrzebne są dwie kopie struktury.

**c) Treść slajdu**
- Ustal próg:
\[
L = 8f_d(n+2)
\]
- Krótki plan:
- `partition all updates into epochs \(E_0,E_1,E_2,\ldots\) of length \(L\)`
- `\(\mathbb A_i\)` is initialized during \(E_{2i-2}\) and \(E_{2i-1}\)
- `\(\mathbb A_i\)` answers queries during \(E_{2i}\) and \(E_{2i+1}\)
- `during \(E_{2i-2}\), spread \(\mathtt{Init}(n,\mathcal K)\) over the whole epoch`
- `during \(E_{2i-1}\), the next structure catches up by processing two updates while the previous one processes one`
- Jeden ważny dopisek:
`Initialization needs uninterrupted access to the old state, so we keep two copies: \(\mathbb A_{i,0}\) active and \(\mathbb A_{i,1}\) as a frozen snapshot.`
- Na dole:
`This is the de-amortization step behind Theorem 5, hence also behind the main theorem.`

**d) Bardzo precyzyjny opis rysunku**
- Pozioma oś czasu z epokami `E_0,E_1,E_2,E_3,\dots`.
- Dwa poziome tory:
- górny: `active structure`
- dolny: `next structure`
- Na dolnym torze dwa wyraźne odcinki: `initialize`, potem `catch up`.
- Obok mały osobny prostokąt podpisany `frozen snapshot copy`.
- Nie więcej niż trzy kolory: aktywna, budowana, snapshot.

**e) Zakotwiczenie źródłowe**
- Przede wszystkim: `main-all.tex`, wspierane przez `main.pdf`.
