* **[Slide 1: Dynamic data structures for twin-ordered matrices]**
  Good morning. I would like to talk about a dynamic data structure for twin-ordered matrices. In particular, I will show how to support both single-cell queries and single-cell flips in ( \log\log n )-type time. This is joint work with my co-authors. Before I get to the result, let me first very quickly set up the object that we want to maintain.

* **[Slide 2: What is the object we want to maintain?]**
  We work with a binary matrix (M) of size (n\times n). The most important point is that the order of rows and columns is fixed, and throughout the whole talk we do not change it. Our data structure is supposed to support two very natural interfaces. The first one is `QueryBit(i,j)`, that is, asking for the value of a specific cell. The second one is `Update(i,j)`, that is, flipping this cell from zero to one or from one to zero. If you look at the highlighted cells in the figure, these are exactly the two operations. However, there is one global condition: after every update, the matrix must still remain (d)-twin-ordered. And this leads us to the question of what exactly such an ordering means.

* **[Slide 3: Twin-ordering by contraction sequences]**
  The definition goes through what is called a contraction sequence. We start from the situation where every row and every column forms its own separate block. Then, step by step, we merge two consecutive row blocks or two consecutive column blocks. At every moment, we look at the zones between the current blocks. Some of them are constant, and some are non-constant, meaning that they contain both zeros and ones. The width of the sequence is the maximum number of such non-constant zones that can touch a single block. A matrix is (d)-twin-ordered if there exists such a sequence of width at most (d). If you look at the successive pictures, the intuition is that globally the structure becomes coarser and coarser, but the local chaos remains bounded. Once we have this model, we can already formulate the actual gap in the literature.

* **[Slide 4: The gap: static yes, dynamic no]**
  There was already a very strong static result. It is known that such a matrix can be represented very compactly, in (O_d(n)) bits, and that one can answer single-cell queries in time (O_d(\log\log n)). But that representation is fundamentally static. That is, it is good once initialized, but not designed for continuous changes of the matrix. So our question is very simple: can we keep similarly fast queries, add updates, and still remain within near-linear memory on word RAM? In other words, can one build a dynamic version of that static picture without losing the most important parameters? The answer is yes, and this is exactly the content of the main theorem.

* **[Slide 5: Main theorem]**
  This is the main result. For fixed (d), if the matrix remains (d)-twin-ordered at all times, then there exists a dynamic data structure using (O_d(n)) memory. It can be initialized from a list of disjoint rectangles representing all ones. Then it answers `QueryBit(i,j)` in (O(\log\log n)) expected worst-case time and performs `Update(i,j)` in the same time. It is worth noticing two nuances. First, the dependence on (d) appears only in memory and initialization. Second, in the dynamic version we pay for flexibility by moving from bits to word-RAM memory. The rest of the talk will not be the full proof, but rather the main idea of why this result is possible. And this idea starts with a very simple geometric object.

* **[Slide 6: Slabs: why near-linear space is possible]**
  This object is a slab, that is, a rectangle completely filled with ones. If you look at the colored regions, each of them is exactly such a slab. The key structural fact says that every (d)-twin-ordered matrix can be described by a family of pairwise disjoint slabs, and there are only (O_d(n)) such slabs. More precisely, we have the bound (|\mathcal K|\le d(2n-2)+1). This is the first reason why near-linear memory can even hope to work. We do not try to remember every one separately. We remember compact rectangles of ones. But an arbitrary such decomposition is still not regular enough for the dynamic part. We first need to normalize it.

* **[Slide 7: Canonical slabs: the normalization step]**
  This is where the canonical slab decomposition appears. Instead of taking arbitrary rectangles of ones, we move to a canonical version. We look column by column at maximal vertical all-one segments, that is, strips. Then we merge into one canonical slab those strips that have the same row interval and persist through consecutive columns. If you look at the figure on the right, you can see that such a version may be more fragmented than some coarser decomposition on the left. But this fragmentation is controlled. We still have only (O_d(n)) canonical slabs. Concretely, the number of these slabs is bounded by an expression based on the number of corners. It is exactly this canonicity that gives us the order needed by the rebuilding algorithm. The next step is to show how to compute such a decomposition efficiently.

* **[Slide 8: Running example, part I: left-to-right sweep for (B_i)]**
  From this point on, I move to one small running example. Please look at the same small matrix and at the table below. The algorithm `Decompose` works by a sweep from left to right. In round (i), we want to understand which vertical strips disappear when we move from column (i) to column (i+1). The sets (O_i) and (C_i) only help identify which fragments are touched by slabs ending here or starting immediately further on. The most important object is (B_i): these are exactly the strips that disappear now. If you look at the highlighted rows of the table and at the red sweep line, you can see that we do not need to recompute everything from scratch. We only care about local changes. This is the first half of the algorithm. In a moment I will show the second half.

* **[Slide 9: Running example, part II: from (A_i,B_i) to canonical slabs]**
  The second half is symmetric. Now we sweep from right to left and obtain the sets (A_i), that is, the strips that appear. The idea is very simple: for every row interval, we remember where the current canonical slab started. This is exactly what the `slab_start` table does. When a given strip appears, we record the start. When it disappears, we already know that we can output the whole slab from the remembered start to the current column. If you look at the lower matrix, this is exactly the result of that process. So we obtain an algorithm that transforms an arbitrary slab decomposition into the canonical slab decomposition in time (O(n+|\mathcal K|\log\log n)). This closes the structural part. Now we still have to turn this into a genuine dynamic data structure.

* **[Slide 10: Amortized structure: (\mathbb P) for slabs, (\mathbb Q) for recent flips]**
  The dynamic data structure has two layers. The layer (\mathbb P) stores the last rebuilt canonical decomposition and serves as a static base for point location. The layer (\mathbb Q) is a small buffer of the newest updates, now stored explicitly at the level of individual cells. If you look at this figure, the colored slabs are in (\mathbb P), while the small overrides sit in (\mathbb Q). A query works in a very natural way: first we check whether the cell is overridden in the buffer, and if not, we ask the base. An update is also natural: we write the new value into the buffer, and on the first touch we read the old value from the base. The key invariant is: (\mathbb Q) always has priority over (\mathbb P). When the buffer becomes too large, we rebuild the base. And this rebuilding is the central point of the whole construction.

* **[Slide 11: Rebuilding (\mathbb P): 0-updates split slabs, uncovered 1-updates become (1\times 1) slabs]**
  During rebuilding, we start from the old slabs in (\mathbb P) and from the buffered changes in (\mathbb Q). Here it is very important to distinguish two types of updates. Zero-updates are the difficult ones, because they split old slabs. These are the ones that force a real change in the geometry. One-updates are easier: if they were not already covered by an old slab, they simply become new (1\times 1) slabs. If you look at the middle panel, you can see exactly this logic: the old rectangle breaks into several untouched and split pieces, and the isolated one becomes a new small slab. From these elements we assemble a family (\mathcal K), which already describes the current matrix, and we get the estimate (|\mathcal K|\le |\mathcal P|+2|\mathcal Q|). And then we simply run `Decompose` again. This gives us the amortized structure. There is only one final step left: how to remove amortization.

* **[Slide 12: From amortized to worst-case: epochs and snapshot copies]**
  Here we use a standard but very important de-amortization. We divide updates into epochs of length (L). One copy of the structure keeps handling the current queries and updates all the time. In parallel, a second copy is prepared in the background. If you look at the lower track of the timeline, during the first epoch we spread the initialization of the new structure across all updates, so that the cost per single update stays small. Then, during the next epoch, this new copy catches up, processing two updates for every one update handled by the old active copy. At the end, it simply takes over. Why do we need a snapshot? Because initialization needs calm, immutable access to the old state. So we keep two copies: the active one, and a frozen snapshot used only for reading. And this is exactly what turns the amortized structure into the expected worst-case structure from the main theorem. What remains now is only the final overall picture.

* **[Slide 13: Take-away]**
  I would like to leave you with one simple chain of ideas. We start from a (d)-twin-ordered matrix. This gives us a small slab decomposition. Then we move to the canonical version, which still has linear size in (n), up to a constant depending on (d). On top of this canonical base, we build two layers: (\mathbb P) for the last rebuilt state and (\mathbb Q) for fresh updates. Finally, we add epochs and snapshot copies in order to obtain not only a good amortized cost, but truly expected worst-case bounds for both operations. So the final message is the following: for (d)-twin-ordered matrices, we have a dynamic data structure with (O_d(n)) memory and (O(\log\log n)) time for both `QueryBit` and `Update`. Thank you very much.
