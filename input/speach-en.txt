* **[Slide 1: Dynamic data structures for twin-ordered matrices]**
  I will talk about a dynamic data structure for d-twin-ordered binary matrices. The goal is to support cell queries and single-cell flips in `O(log log n)` expected worst-case time with near-linear memory. Let me start with the object.

* **[Slide 2: What is the object we want to maintain?]**
  The object is a binary `n by n` matrix `M`, with one crucial restriction: the order of rows and columns never changes. We are not permuting anything on the fly. On top of that fixed matrix, we want two operations. `QueryBit(i,j)` asks for one cell, and `Update(i,j)` flips one cell. The marked cells in the picture show exactly that interface. The only global promise is that after every update, the matrix is still d-twin-ordered.

* **[Slide 3: Twin-ordering by contraction sequences]**
  Let me unpack that promise just enough for the rest of the talk. A contraction sequence starts with singleton row blocks and singleton column blocks. Then we repeatedly merge two consecutive row blocks or two consecutive column blocks. At every stage, the current blocks define zones of the matrix. Some zones are constant, and some are mixed. The width is bounded by `d` if every current row block and every current column block sees at most `d` mixed zones. The snapshots on the slide show the idea: the partition gets coarser, but local irregularity stays bounded. A matrix is d-twin-ordered when such a sequence exists.

* **[Slide 4: The gap: static yes, dynamic no]**
  There was already a strong static result for this setting. A d-twin-ordered matrix can be stored in `O_d(n)` bits, and single-entry queries can be answered in `O_d(log log n)` time. But that representation is fundamentally static. It is not designed for a stream of flips. So the question is whether we can keep the same flavor of efficiency in the dynamic world: support `QueryBit` and `Update`, keep the d-twin-ordered promise, and still use only near-linear memory on word RAM. That is the gap this paper closes.

* **[Slide 5: Main theorem]**
  Here is the theorem in spoken form. Fix `d`, and consider a binary matrix that is updated over time, always remaining d-twin-ordered. Then there is a dynamic data structure with `O_d(n)` memory that can be initialized from a list `K` of disjoint all-one rectangles, answers `QueryBit(i,j)` in `O(log log n)` expected worst-case time, and performs `Update(i,j)` in the same expected worst-case time. Two nuances matter. First, the dependence on `d` sits only in initialization and memory; the update and query bounds themselves do not carry `d`. Second, the dynamic structure uses word-RAM memory, so this is `O_d(n log n)` bits rather than the bit-optimal static bound. The rest of the talk is the mechanism behind that theorem.

* **[Slide 6: Slabs: why near-linear space is possible]**
  The first structural object is a slab: a rectangle filled entirely with ones. A slab decomposition is then a family `K` of pairwise disjoint slabs that covers every one in the matrix. The key point is not just the definition. The key point is compression. We are not storing ones cell by cell. We are storing large monochromatic rectangles, and for d-twin-ordered matrices there are only linearly many of them. The bound on the slide says that `|K|` is at most `d(2n-2)+1`. So near-linear space becomes plausible already at this level. The figure is useful here because it shows what the description really looks like: a few rectangles encode the whole set of ones. But for updates, an arbitrary slab decomposition is still too loose. We need a normalized one that behaves predictably across columns.

* **[Slide 7: Canonical slabs: the normalization step]**
  That normalization is the canonical slab decomposition. An `i`-strip is a maximal vertical all-one segment inside column `i`. So strips are the atomic vertical pieces you see when you scan the matrix one column at a time. If the same row interval survives through consecutive columns, those strips are siblings, and one sibling class becomes one canonical slab. So the canonical decomposition follows a strip exactly as long as it persists, and no longer. The right panel is more fragmented than the left one, but in a disciplined way. Why is it still only `O_d(n)`? Because every internal canonical slab must witness a local obstruction, called a corner, near the place where it stops being extendable. There are only linearly many such witnesses, up to a constant depending on `d`, and the slabs touching the boundary contribute only another `4n`. That is why canonicity does not destroy linear size.

* **[Slide 8: Running example, part I: left-to-right sweep for (B_i)]**
  Now let me use the small `4 by 4` example and turn it into an actual algorithm. The left-to-right sweep computes the sets `B_i`, meaning the strips that are present in column `i` but disappear before column `i+1`. The red sweep line tells you which round we are in. The sets `O_i` and `C_i` are only bookkeeping: `O_i` lists strips opened by slabs that start in column `i`, and `C_i` lists strips closed by slabs that end in column `i`. The live object is the current set of strips in the present column, and round `i` updates it from `strips_i` to `strips_{i+1}`. The important insight is that only locally affected strips matter: strips containing something from `C_i`, or touching something that opens in `O_{i+1}`. Everything else survives unchanged. Notice the highlighted rows of the table. In round `1`, the two one-cell strips disappear immediately, so they belong to `B_1`. In round `3`, the long strip `[2,4]` is the one that vanishes, so it is exactly `B_3`.

* **[Slide 9: Running example, part II: from (A_i,B_i) to canonical slabs]**
  The second sweep is symmetric and gives the sets `A_i`, the strips that appear when we enter column `i`. Once we have both `A_i` and `B_i`, the remaining step is simple. We scan left to right once more and keep an array called `slab_start`. For each row interval, it remembers where the current canonical slab began. Watch the top table. At `i = 2`, the strip `[2,4]` appears, so we record start `2` for that interval. At `i = 3`, the same interval now belongs to `B_3`, so we output the slab running from column `2` through column `3`. At `i = 4`, two new intervals start and also end there, so they become one-column canonical slabs. The lower matrix is the final output of this bookkeeping. So the story from slides 8 and 9 is complete: first identify what disappears and what appears, then match each start with the corresponding end. That is exactly how `Decompose` turns any slab decomposition into the canonical one.

* **[Slide 10: Amortized structure: (P) for slabs, (Q) for recent flips]**
  With canonical slabs in hand, we can finally make the structure dynamic. The data structure has two layers with one clean invariant. `P` stores the canonical slab decomposition from the last rebuild, through static two-dimensional point location. So `P` is the stale but structured base. `Q` stores the newer updates explicitly, as a dictionary on cells. So `Q` is the fresh but small override layer. A query first asks `Q`. If the cell was touched since the last rebuild, `Q` already contains the truth and has priority over everything else. Only on a miss do we ask `P` whether the cell lies in one of the stored base slabs. An update is also local: if the cell is already in `Q`, we just flip the stored value; otherwise we recover its old value from `P`, negate it, and insert that value into `Q`. The reason `Q` must override `P` is that `P` intentionally lags behind reality. We keep doing this until `Q` reaches the threshold `8 f_d(n+2)`, and only then do we rebuild the base.

* **[Slide 11: Rebuilding (P): 0-updates split slabs, uncovered 1-updates become (1x1) slabs]**
  This rebuilding step is the technical heart of the amortized structure, so let me slow down here. We start with the old slabs stored in `P` and with the explicit updates stored in `Q`. The first split is conceptual: separate `Q` into zero-updates and one-updates. The hard part is the zero-updates, because only they can destroy old geometry. A zero-update landing inside an old all-one slab carves a hole in something that used to be a solid rectangle. So we first assign every zero-update to the old slab that contains it. Now fix one affected slab and sort its zero-updates by row, then by column. We process them row by row. Everything above the next touched row is still untouched, so that whole prefix can be emitted immediately as one intact slab. Then we isolate the single-row slice at the touched row. On that row, we cut only at the updated columns and keep the remaining intervals as smaller one-row slabs. After that, we continue with the suffix below. So the slab is not shattered cell by cell. It is cut only where the zero-updates force cuts, and the algorithm moves downward through the affected rows. That is the row-by-row mechanism hidden behind the picture in the middle panel.

  The one-updates are easier. Once the zero-updates have removed the cells that should become zero, we inspect each buffered one-update. If that cell was already covered by an old slab, then nothing new has to be added. If it was uncovered, then it simply becomes a new `1x1` slab. At the end, the family `K` consists of untouched old slabs, split pieces from affected slabs, and uncovered `1x1` slabs from one-updates. This already is a valid slab decomposition of the current matrix. The crucial estimate is `|K| <= |P| + 2|Q|`, so rebuilding grows only linearly with the amount of buffered change. Once `K` is ready, we run `Decompose` again and recover a new canonical base.

* **[Slide 12: From amortized to worst-case: epochs and snapshot copies]**
  One last gap remains. The rebuild we have just discussed is cheap only in the amortized sense. To remove that amortization, we schedule the rebuild work across epochs. Let `L = 8 f_d(n+2)`. This is exactly the buffer threshold from the previous slide, now reinterpreted as the epoch length. Partition the updates into epochs of length `L`. During one preparatory epoch, one copy of the structure stays fully active: it answers queries and processes the real updates online. In parallel, a second copy is initialized in the background from a read-only snapshot of the old state. The total initialization cost is `O(L log log n)`, so if we spread it across those `L` updates, we pay only `O(log log n)` extra per update. By the end of that epoch, the new copy exists, but it is one whole epoch behind.

  The next epoch is the catch-up phase. The old active copy continues to process the real stream, one update at a time. Meanwhile, the new copy replays two pending updates for every one real update handled above. Since it starts exactly `L` updates behind and gets `L` time slots in this epoch, two-for-one is enough to erase the lag completely by the end. Then the new copy takes over as the active structure. The subtle point is the snapshot. We need it because initialization requires coherent, uninterrupted access to the old `P` and `Q`. The live structure cannot offer that while incoming updates keep modifying it. The frozen snapshot gives a stable past state to rebuild from, while the active copy keeps the system online. That is why the amortized bound becomes an expected worst-case one.

* **[Slide 13: Take-away]**
  Let me close with the full pipeline in one sentence. Twin-ordering gives us a linear-size slab description, canonicalization turns that description into something algorithmically stable, the two-layer view `P + Q` absorbs recent changes locally, and epochs with snapshot copies remove amortization. Putting these pieces together, we get a genuinely dynamic data structure for d-twin-ordered matrices with `O_d(n)` memory and `O(log log n)` expected worst-case time for both `QueryBit` and `Update`. That is the main message I wanted to leave you with. Thank you, and I will be happy to take questions.
