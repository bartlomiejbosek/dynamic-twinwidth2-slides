3. **a) Tytuł**
   **Twin-ordering by contraction sequences**

   **b) Cele slajdu**

   * Dać minimalną, ale prawdziwą definicję (d)-twin-ordered matrix.
   * Użyć figury z artykułu zamiast wymyślać nową ilustrację.
   * Doprowadzić słuchacza do punktu: „rozumiem model”.

   **c) Treść slajdu**

   * Krótka definicja:

     * repeatedly merge two consecutive row blocks or two consecutive column blocks,
     * each row/column block may see only few **non-constant zones**,
     * width (d) means: at every step, at most (d) such zones per block.
   * Wyróżniona linia:
     **(M) is (d)-twin-ordered if it admits such a contraction sequence.**
   * Mały dopisek:
     **Throughout the paper, the order is fixed.**

   * Wszystkie non-constant zones zaznaczyć jasnofioletowym tłem jak w artykule.
   * Pod każdym stanem mały podpis: `merge rows`, `merge columns`, `width stays 2 in the example`. 

4. **a) Tytuł**
   **The gap: static yes, dynamic no**

   **b) Cele slajdu**

   * Osadzić pracę względem poprzedniego wyniku.
   * Sformułować dokładnie pytanie badawcze.
   * Przygotować natychmiastowe wejście w main theorem.

   **c) Treść slajdu**

   * Lewa kolumna:
     **Static result [Pilipczuk–Sokołowski–Zych-Pawlewicz]**

     * (O_d(n)) bits
     * single-entry queries in (O_d(\log\log n))
     * no efficient updates
   * Prawa kolumna:
     **Our question**

     * can we support `QueryBit` and `Update`
     * while keeping near-linear space
     * under the promise that the matrix always remains (d)-twin-ordered?
   * Na dole jedna linia:
     **This is exactly the dynamic problem addressed in the paper.**

   **d) Bardzo precyzyjny opis rysunku**

   * Dwie proste karty obok siebie.
   * Lewa karta: ikona „zatrzaśniętej” macierzy, podpis `static`.
   * Prawa karta: ta sama macierz z dwiema strzałkami update, podpis `dynamic ?`.
   * Między nimi gruba strzałka w prawo.
   * Bez dodatkowych ozdobników. 

5. **a) Tytuł**
   **Main theorem**

   **b) Cele slajdu**

   * Ustawić wszystko, co dalej będzie tłumaczone.
   * Dać publiczności jedną rzecz do zapamiętania.

   **c) Treść slajdu**

   * Duży theorem box:
     [
     \textbf{Dynamic DS for } d\text{-twin-ordered } M\in{0,1}^{n\times n}
     ]

     * `Init(n,K)` in (O_d((n+|K|)\log\log n))
     * `QueryBit(i,j)` in (O(\log\log n)) expected worst-case time
     * `Update(i,j)` in (O(\log\log n)) expected worst-case time
     * memory (O_d(n))
   * Pod theorem boxem dwa krótkie dopiski:

     * (K) = list of disjoint all-1 rectangles
     * dependence on (d) appears only in memory and initialization

   **d) Bardzo precyzyjny opis rysunku**

   * Bez osobnej figury.
   * Główna treść slajdu to centralny theorem box.
   * Pod spodem mała tabelka `static vs dynamic` z trzema wierszami: space / query / update.
   * Komórka `update` dla wyniku statycznego pusta lub z kreską, dla nowego wyniku pogrubione (O(\log\log n)).

   **b) Cele slajdu**

7. **a) Tytuł**
   **Canonical slabs: the normalization step**

   **b) Cele slajdu**

   * Wyjaśnić, że „jakiś” slab decomposition nie wystarcza.
   * Wprowadzić canonical slab decomposition jako właściwy obiekt do utrzymywania.
   * Pokazać, że kanoniczność nie niszczy liniowego rozmiaru.

   **c) Treść slajdu**

   * Trzy krótkie definicje:

     * **(i)-strip** = maximal all-1 segment in column (i)
     * **siblings** = same row interval persisting through consecutive columns
     * **canonical slab** = union of one sibling class
   * Wyróżniona linia:
     **Each strip belongs to exactly one canonical slab.**
   * Potem:
     [
     |\mathcal R|\le 4f_d(n+2)+4n
     ]
   * Krótki komentarz:
     **Canonical slabs are still only (O_d(n)).**

   **d) Bardzo precyzyjny opis rysunku**

   * Nad prawą podpis `canonical slab decomposition`.

   * Ważne, by wzrokowo było jasne: kanoniczna wersja jest bardziej „rozszczepiona”, ale nadal ma kontrolowany rozmiar.

10. **a) Tytuł**
    **Amortized structure: (P) for the past, (Q) for recent flips**

    **b) Cele slajdu**

    * Przejść od reprezentacji do dynamicznej struktury danych.
    * Wyjaśnić invariant dwóch warstw.
    * Uczynić `QueryBit` i `Update` intuicyjnymi.

    **c) Treść slajdu**

    * Dwa duże boksy:

      * **(P)**: static 2D orthogonal point location on the last canonical slab decomposition
      * **(Q)**: hash table of recent updates
    * Invariant:
      [
      (i,j)\in Q \Rightarrow M[i,j]=Q[i,j]
      ]
      [
      (i,j)\notin Q \Rightarrow M[i,j]=1 \iff (i,j)\text{ lies in a slab of }P
      ]
    * Dwa krótkie punkty:

      * `QueryBit`: check (Q), otherwise point-location in (P)
      * `Update`: flip in (Q); rebuild only when (Q) becomes too large

    **d) Bardzo precyzyjny opis rysunku**

    * Jedna macierz jako tło.
    * Kilka dużych półprzezroczystych prostokątów oznaczonych `stored in P`.
    * Na wierzchu 3–4 pojedyncze komórki z grubym obramowaniem oznaczone `stored in Q`.
    * Po lewej mała strzałka `QueryBit(i,j): Q first, then P`.
    * Po prawej mała strzałka `Update(i,j): only Q for now`.

12. **a) Tytuł**
    **From amortized to worst-case: epochs and two copies**

    **b) Cele slajdu**

    * Domknąć ostatni brak: dlaczego update jest worst-case, a nie tylko amortized.
    * Pokazać prostą ideę epok.
    * Wyjaśnić, po co potrzebne są dwie kopie struktury.

    **c) Treść slajdu**

    * Krótki plan:

      * let (L=8f_d(n+2))
      * partition updates into epochs of length (L)
      * build the next structure during epoch (E_{2i-2})
      * catch up during (E_{2i-1})
      * answer queries during (E_{2i}) and (E_{2i+1})
    * Potem jeden ważny punkt:
      **Need two copies: one active, one frozen snapshot for the next initialization.**
    * Na dole:
      **This yields (O(\log\log n)) expected worst-case update time.**

    **d) Bardzo precyzyjny opis rysunku**

    * Pozioma oś czasu z epokami (E_0,E_1,E_2,E_3,\dots)
    * Dwa poziome tory:

      * górny: `active structure`
      * dolny: `next structure`
    * Na dolnym torze dwa stany: `initialize`, potem `catch up`.
    * Obok mały osobny prostokąt podpisany `snapshot copy`.
    * Nie więcej niż trzy kolory: aktywna, budowana, snapshot.

16. **a) Tytuł**
    **Backup: rebuilding (K) from (P) and (Q) in detail**

    **b) Cele slajdu**

    * Odpowiedzieć na pytanie o techniczne detale recomputation.
    * Rozbić procedurę na kilka prostych kroków.

    **c) Treść slajdu**

    * Kroki:

      1. sort (Q) lexicographically
      2. assign 0-updates to slabs of (P)
      3. split each affected slab row by row
      4. add (1\times 1) slabs for uncovered 1-updates
      5. obtain (|K|\le |P|+2|Q|)
      6. run `Decompose(n,K)`
    * Na dole:
      **This is exactly the step amortized over the threshold (L).**

    **d) Bardzo precyzyjny opis rysunku**

    * Flowchart z pięcioma prostokątami połączonymi strzałkami:
      `old P` → `sort Q` → `split on 0-updates` → `add 1×1 slabs` → `Decompose` → `new P`
    * Obok mały przykład jednego dużego slaba rozcinanego przez dwa 0-updates w tym samym wierszu.
    * Bez dekoracji, tylko techniczna czytelność.
